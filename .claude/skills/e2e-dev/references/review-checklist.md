# Code Review 检查清单

在 Phase 4（审核与修复）阶段使用此清单逐项检查。每项标记 ✅ 通过 / ❌ 不通过 / ⏭️ 不适用。

---

## 1. 代码质量

| #   | 检查项            | 说明                                       |
| --- | ----------------- | ------------------------------------------ |
| 1.1 | 命名清晰          | 变量、函数、类型名能自解释，不用缩写       |
| 1.2 | 函数职责单一      | 每个函数只做一件事，不超过 50 行           |
| 1.3 | 无重复代码        | 相同逻辑已提取为公共函数/工具              |
| 1.4 | 错误处理完整      | 异步操作有 try-catch，API 调用有错误码处理 |
| 1.5 | 无魔法数字/字符串 | 常量已提取并命名                           |
| 1.6 | 类型安全          | 无 `any` 滥用，泛型使用合理                |

## 2. 架构与设计

| #   | 检查项       | 说明                                                               |
| --- | ------------ | ------------------------------------------------------------------ |
| 2.1 | 符合项目分层 | 遵循 `src/` 下的模块划分（pipeline / publish / noteatom / shared） |
| 2.2 | 无不当耦合   | 新代码没有反向依赖（如 shared 模块依赖 publish）                   |
| 2.3 | 接口设计合理 | 导出的 API 签名稳定、向后兼容                                      |
| 2.4 | 依赖合理     | 没有引入不必要的第三方依赖                                         |
| 2.5 | 副作用可控   | 文件 I/O、网络请求等副作用集中在明确的位置                         |

## 3. 测试覆盖

| #   | 检查项         | 说明                                             |
| --- | -------------- | ------------------------------------------------ |
| 3.1 | 验收条件全覆盖 | Phase 1 拆解的每条 AC 都有对应测试               |
| 3.2 | 边界情况       | 空输入、异常输入、边界值都有测试                 |
| 3.3 | 测试独立       | 每个测试用例不依赖其他用例的执行顺序             |
| 3.4 | 测试可读       | 测试名描述了行为（`should do X when Y`）         |
| 3.5 | 无硬编码路径   | 测试中使用 `tmp` 目录或 mock，不依赖本机绝对路径 |

## 4. 安全性

| #   | 检查项        | 说明                                      |
| --- | ------------- | ----------------------------------------- |
| 4.1 | 无 PII 泄露   | 代码中无本机绝对路径、API Key、Token      |
| 4.2 | `.env` 未追踪 | `.env` 在 `.gitignore` 中，未被 `git add` |
| 4.3 | 输入校验      | 外部输入（文件路径、用户参数）有校验      |
| 4.4 | 依赖安全      | 新依赖无已知漏洞（`npm audit`）           |

## 5. 可维护性

| #   | 检查项     | 说明                                     |
| --- | ---------- | ---------------------------------------- |
| 5.1 | 注释有价值 | 复杂逻辑有注释说明"为什么"，而非"做什么" |
| 5.2 | 文档同步   | 新增的公共 API 有 JSDoc / README 说明    |
| 5.3 | 日志适当   | 关键路径有日志输出，便于调试             |
| 5.4 | 配置外置   | 可变参数通过配置/环境变量传入，非硬编码  |

## 6. 提交规范

| #   | 检查项       | 说明                                          |
| --- | ------------ | --------------------------------------------- |
| 6.1 | 提交信息格式 | 符合 `feat:` / `fix:` / `test:` 等前缀规范    |
| 6.2 | 提交粒度合理 | 一个提交完成一个逻辑变更，不混杂无关改动      |
| 6.3 | 无调试残留   | `console.log`、`debugger`、注释掉的代码已清理 |

---

## 审核结论模板

```markdown
## Code Review 结果

### 通过项

- ✅ 1.1 命名清晰
- ✅ 1.2 函数职责单一
- ...

### 问题项

- ❌ 3.2 缺少边界测试：未测试空数组输入
- ❌ 4.1 路径泄露：`src/foo.ts:42` 包含 `/Users/xxx/`

### 风险评估

- **低风险**：无阻塞问题
- **中风险**：有 1-2 个需修复的问题
- **高风险**：有架构或安全阻塞问题

### 建议

- 修复以上 ❌ 项后重新提交
- 或确认 ⏭️ 项为不适用
```
